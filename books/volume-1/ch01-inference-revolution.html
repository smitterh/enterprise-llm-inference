Chapter 1 — The Inference Revolution

Author

Hugo Smitter

1 Chapter 1 — The Inference Revolution

1.1 Why This Matters

Every major shift in computing has created a new engineering discipline.

The rise of relational databases created database administrators and database engineers. The growth of the Internet produced web engineers. Cloud computing gave rise to cloud architects, site reliability engineering (SRE), and platform engineering. Kubernetes introduced an entirely new ecosystem of cloud-native infrastructure specialists.

Large Language Models (LLMs) represent another such shift.

While much public attention has focused on training increasingly capable models, the overwhelming majority of organizations will never train a frontier model. Instead, they will deploy, serve, secure, monitor, and optimize models that already exist. The challenge is no longer how to create a model—it is how to run one efficiently, economically, and reliably at scale.

This challenge is known as LLM inference.

Inference engineering sits at the intersection of distributed systems, GPU computing, networking, scheduling, storage, cloud-native platforms, and machine learning infrastructure. It is becoming its own engineering discipline because traditional software architectures were never designed to execute probabilistic models that generate text one token at a time.

This book is about that discipline.

2 1.1 Computing Is Entering the Inference Era

For more than three decades, enterprise software has been dominated by transactional systems.

A typical application processed requests using a familiar pipeline:

flowchart LR

A[Client] --> B[Load Balancer]
B --> C[Application]
C --> D[Database]
D --> C
C --> A

Although the technologies evolved—from monoliths to microservices, from virtual machines to containers, from data centers to cloud-native platforms—the underlying execution model remained largely unchanged.

A request arrived.

Business logic executed.

Data was retrieved.

A response was returned.

Most architectural improvements focused on scaling this workflow. Faster databases, better caches, improved load balancers, more resilient clusters, and horizontal scaling all targeted the same basic request/response model.

Large Language Models fundamentally change that model.

Instead of retrieving information from a database, an LLM synthesizes new information by executing billions or even trillions of mathematical operations across neural network layers.

Instead of returning a complete response immediately, it generates one token at a time.

Instead of consuming mostly CPU resources, inference depends primarily on accelerators such as GPUs.

Instead of milliseconds of computation, requests may execute for several seconds while continuously consuming expensive hardware.

The architectural implications are profound.

The question is no longer:

“How do I scale my web service?”

Instead, organizations must answer questions such as:

  • How do I keep GPUs fully utilized?
  • How many concurrent requests can a model serve?
  • Where should prompts be routed?
  • How is conversational context preserved?
  • How do I minimize latency while maximizing throughput?
  • How do I reduce inference cost without sacrificing quality?

These questions define a new category of systems engineering.

3 1.2 Training Is Not the Business Problem

Popular media often portrays artificial intelligence as a race to build larger and larger models.

Companies announce trillion-parameter models.

Research papers focus on training techniques.

Benchmark leaderboards celebrate increasingly capable systems.

This creates the impression that training is the central engineering challenge.

For most enterprises, it is not.

Training a frontier model requires enormous investments in hardware, specialized expertise, and months of continuous computation. Only a small number of organizations possess the resources necessary to train state-of-the-art foundation models.

Most enterprises operate very differently.

They consume pre-trained models from providers such as OpenAI, Anthropic, Google, Meta, Mistral AI, or open-source communities. Their competitive advantage lies not in creating models, but in integrating them into products and business processes.

Consider several common enterprise applications:

Application Primary Workload
Customer support Millions of inference requests
Coding assistants Continuous inference
Enterprise search Retrieval plus inference
Document summarization Batch inference
Fraud analysis Real-time inference
AI agents Multi-step inference

None of these applications train models during normal operation.

They perform inference.

Every user interaction becomes another inference request.

An organization serving one million users may execute billions of inference requests every month while never training a single neural network.

This distinction changes everything.

Training is typically performed once.

Inference executes continuously.

Training is a capital investment.

Inference becomes an operational expense.

Training determines what a model can know.

Inference determines whether a product is economically viable.

As models become commoditized, inference increasingly becomes the competitive advantage.

Organizations that can deliver lower latency, higher throughput, better reliability, and lower operating costs gain a measurable business advantage, even when using the same underlying model as their competitors.

4 1.3 Every Token Has a Cost

Traditional software scales primarily with CPU, memory, storage, and network bandwidth.

Inference systems introduce a different economic model.

Every generated token consumes computational resources.

Those resources have direct financial cost.

A simple prompt such as:

“Summarize this five-page document.”

initiates a surprisingly complex execution pipeline.

The request must be tokenized.

The model performs an initial computation over the entire prompt.

GPU memory is allocated.

Intermediate state is stored.

The decoder generates one token.

Then another.

And another.

Each token requires additional computation until the response completes.

Unlike a REST API that often completes after a single database query, inference workloads remain active throughout the entire generation process.

Longer conversations consume more GPU time.

Longer outputs require more decoding.

More concurrent users require more accelerator capacity.

The result is an entirely different optimization problem.

Instead of asking,

“Can this server handle more requests?”

platform engineers begin asking,

“Can this GPU generate more tokens per second?”

This seemingly small shift in perspective drives nearly every architectural decision discussed throughout the remainder of this book.

5 1.4 Inference Is Not Just Another API

At first glance, invoking an LLM appears deceptively simple. An application sends a prompt to a REST or gRPC endpoint and receives a response. From the perspective of the client, the interaction resembles any other service call.

Internally, however, the request follows a very different execution path.

A traditional web application typically performs a small amount of computation before retrieving or updating information in a database.

flowchart LR

Client --> LoadBalancer["Load Balancer"]
LoadBalancer --> Service
Service --> Database
Database --> Service
Service --> Client

The duration of the request is usually determined by network latency, business logic, and database performance.

Inference systems introduce an entirely different pipeline.

flowchart LR

Client --> Gateway
Gateway --> Router
Router --> Tokenizer
Tokenizer --> Scheduler
Scheduler --> ModelServer["Model Server"]
ModelServer --> GPU
GPU --> Decoder
Decoder --> Stream
Stream --> Client

Several important differences immediately emerge.

First, inference requests are stateful while they execute. The system must remember every token that has already been generated.

Second, the request remains active until generation completes. Unlike a database query that returns all results at once, an LLM produces output incrementally.

Third, expensive accelerator resources remain occupied for the lifetime of the request.

Finally, every additional generated token requires another iteration through the decoding process.

These characteristics explain why traditional load balancing techniques alone are insufficient for inference workloads. Two requests with identical arrival times may consume vastly different amounts of GPU time depending on prompt length, response length, and model complexity.

Inference systems therefore require intelligent scheduling rather than simple request distribution.

6 1.5 Why Traditional Scaling Stops Working

Cloud-native engineers are accustomed to solving performance problems by adding replicas.

If a web service experiences higher demand, Kubernetes schedules additional Pods. A load balancer distributes requests across them, and throughput increases almost linearly.

Inference systems rarely behave this way.

Several new constraints dominate system performance.

6.1 GPU Memory

The largest models require tens or hundreds of gigabytes of accelerator memory.

Even when computation is available, insufficient memory prevents additional requests from executing concurrently.

Memory—not compute—is often the limiting resource.

6.2 Model Loading

Loading a large language model into GPU memory may take tens of seconds.

Creating new replicas on demand is therefore much slower than scaling a stateless application container.

Inference platforms frequently keep models resident in memory to avoid repeated loading costs.

6.3 Context Length

Longer conversations require more computation and more memory than short prompts.

Two users making requests to the same model may consume dramatically different resources.

Capacity planning becomes probabilistic rather than deterministic.

6.4 Continuous Batching

Modern inference engines improve throughput by combining requests that arrive at different times into a shared execution batch.

This technique, known as continuous batching, dramatically improves GPU utilization but introduces sophisticated scheduling algorithms that have no equivalent in traditional web applications.

6.5 KV Cache

Every generated token creates intermediate state that must remain available while the conversation continues.

This state, known as the KV cache, often becomes the dominant consumer of GPU memory.

As request concurrency increases, cache management becomes as important as computation itself.

These constraints explain why inference systems require specialized software such as vLLM, SGLang, TensorRT-LLM, and llm-d.

Their purpose is not simply to execute models.

Their purpose is to maximize useful work performed by expensive accelerator hardware.

7 1.6 A New Engineering Discipline

Every significant shift in computing eventually produces new abstractions, new tools, and new specialists.

Relational databases produced database engineering.

Cloud computing produced cloud architecture.

Container orchestration produced platform engineering.

Inference systems are following the same trajectory.

Modern inference engineers routinely work across disciplines that historically had little overlap.

They must understand distributed systems well enough to design resilient serving platforms.

They must understand GPU hardware well enough to recognize memory bottlenecks.

They must understand networking because every millisecond of latency affects user experience.

They must understand Kubernetes because most production inference platforms are deployed on cloud-native infrastructure.

They must understand machine learning systems—not to train models—but to deploy, scale, route, monitor, and optimize them.

This combination of skills rarely exists within a single traditional engineering role.

As organizations deploy larger numbers of models, inference engineering is emerging as a specialized discipline focused on four primary objectives:

  • Deliver low latency.
  • Maximize accelerator utilization.
  • Reduce operating cost.
  • Maintain reliability at production scale.

Throughout this book we will examine the technologies, architectures, and design patterns that enable these objectives.

8 1.7 What You Will Learn

This volume establishes the conceptual foundations of inference engineering.

The remaining chapters explain:

  • how text becomes tokens;
  • how transformer models execute during inference;
  • why prefill and decode exhibit different performance characteristics;
  • why the KV cache is central to modern inference systems;
  • which metrics accurately describe inference performance; and
  • why GPUs dominate large-scale model serving.

Later volumes build upon these foundations to explore model servers, schedulers, routing, inference gateways, Kubernetes deployment patterns, distributed serving, observability, benchmarking, and enterprise architectures.

By the end of this series, you should be able to reason about an inference platform in the same way experienced cloud architects reason about microservice platforms today.

9 Chapter Summary

The rapid adoption of large language models is transforming enterprise computing.

Although public attention often focuses on model training, most organizations derive value through inference. Every interaction with an AI assistant, coding copilot, document summarizer, or enterprise agent becomes an inference request competing for expensive accelerator resources.

Unlike traditional web applications, inference systems must optimize GPU memory, scheduling, batching, and token generation while balancing latency, throughput, and cost.

These challenges have given rise to a new engineering discipline focused on designing efficient, reliable, and scalable inference platforms.

The remaining chapters build the knowledge required to understand and engineer those systems.

10 Key Takeaways

  • Most enterprises perform inference rather than training.
  • Inference workloads differ fundamentally from traditional web services.
  • GPUs are valuable because inference is computationally intensive.
  • GPU memory is often more constrained than GPU compute.
  • Efficient scheduling is as important as fast hardware.
  • Inference engineering combines cloud-native systems engineering with machine learning infrastructure.

11 Glossary

Term Definition
Inference Executing a trained model to produce predictions or generated output.
Token The smallest unit of text processed by a language model.
GPU Graphics Processing Unit, optimized for massively parallel computation.
Continuous Batching Dynamically combining requests to maximize accelerator utilization.
KV Cache Stored attention keys and values reused during token generation.
Latency Time required to produce a response.
Throughput Amount of work completed over time, often measured as tokens per second.

13 Hands-on Lab 1 — Your First Inference Server

13.1 Objective

Observe the complete lifecycle of an inference request.

13.2 Steps

  1. Install Ollama.
  2. Download a lightweight open model.
  3. Start the model locally.
  4. Send several prompts of varying length.
  5. Observe streamed token generation.
  6. Measure:
    • total response time;
    • approximate tokens per second; and
    • CPU or GPU utilization during execution.

13.3 Questions

  • Does a longer prompt always produce a longer response?
  • Which appears to take longer: generating the first token or generating subsequent tokens?
  • What hardware resource becomes most heavily utilized?
